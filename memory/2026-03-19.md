# 2026-03-19 记录

## 数据存储架构分析

### OpenClaw 与 iFlow 的数据关系：

1. **OpenClaw 本地存储**：
   - 会话上下文管理
   - 配置文件和设置
   - 技术文档和日志
   - 智能网关处理数据

2. **iFlow 模型处理**：
   - 实际的对话内容
   - AI 模型的响应生成
   - 模型的上下文记忆
   - 语义理解和处理

### 聊天记录流向：

```
用户输入 → OpenClaw (本地处理) → iFlow SDK → iFlow 模型 → AI响应 → OpenClaw → 用户
                                    ↓
                              聊天记录存储
```

### 关键发现：

- **OpenClaw**：作为网关和界面，处理用户交互
- **iFlow**：作为 AI 模型，处理语义和生成响应
- **存储位置**：聊天记录主要存储在 iFlow 服务器端
- **本地缓存**：OpenClaw 保存会话上下文，但不保存完整聊天记录

### 用户身份问题：

- 切换到 iFlow 模型时，底层仍然是 OpenClaw 在服务
- 显示的"心流 CLI"是 iFlow 模型的预设身份
- 已修改配置，让 iFlow 模型也显示为"灵犀"

### 结论：

用户与 iFlow 的聊天记录主要存储在 iFlow 服务器端，OpenClaw 本地主要保存会话上下文和技术配置。
